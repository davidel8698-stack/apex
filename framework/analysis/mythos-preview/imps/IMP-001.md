## Ecosystem analysis — 10 שאלות חובה

### 1. מטרת הרחיב (Purpose)
[`framework/agents/critic.md`](../../../agents/critic.md) הוא **הסקירה הקלין-רום היריבית** של APEX. הוא רץ אחרי executor, לפני verifier, וזה ה־choke point היחיד שבו טענות (RESULT.json) ומציאות (diff + filesystem + git) מושוות **ללא זיהום מסיפור ה־executor**. הוא עוגן האמון — גם אם executor משקר ב־RESULT.json, ה־critic מאמת מחדש מול הקבצים/diff/git. כיום מבצע אימות filesystem (קובץ קיים, בדיקות עוברות) ואימות commit (SHA + הודעה).

### 2. למה דווקא כאן (Architectural justification)
- **לפני critic**: ה־executor עצמו אינו אובייקטיבי — הוא הצד הנבדק.
- **אחרי critic** (verifier, phase-tag): המשימה כבר נחשבת "done"; קשה יותר לחזור אחורה ובמובן ה־UX זה משקיף "מאשים בדיעבד".
- **PreToolUse hooks** (destructive-guard, path-guard): רואים פקודות יחידות, לא RESULT.json אגרגי — לא יכולים להצליב בין `files_modified[]` המוצהר לבין מה ש־git בפועל מכיר.
- **clean-room protocol** ב־critic הוא **הוא המקום היחיד** שבו ניתן לבדוק "האם RESULT.json תואם את מציאות git" בלי שה־executor יסביר את עצמו מתוך הניתוח. זה קריטי — Mythos F-411-08 הראה ש־"executor verbalized reasoning יכול להיות נקי בזמן שהייצוג הפנימי מטעה".

### 3. מה לא תקין כרגע (עובדה)
[`framework/agents/critic.md`](../../../agents/critic.md) שורות 29-35 (סעיף "FILESYSTEM-LEVEL VERIFICATION") כולל:
- "Claimed file created → `ls` that file" ✓
- "Claimed tests pass → re-run the verify_command" ✓
- "Claimed commit → `git log --oneline -1` and confirm SHA + message" ✓ — **אבל זה רק לטענות commit מפורשות**

**מה חסר**: עבור כל ערך ב־`files_modified[]` ב־RESULT.json, אין צ׳ק שהקובץ מופיע ב־`git log --since=<task_start>` **או** ב־`git diff HEAD` **או** ב־`git status --porcelain`. כתוצאה — אם executor כתב לקובץ דרך bypass של git (לדוגמה עריכת `.git/index` ישירות, הזרקת `core.fsmonitor`, או `LD_PRELOAD` למלכודת syscalls), הקובץ עשוי להיות שונה ממה ש־git רואה, **ו־critic לא יזהה את הפער**.

### 4. שורש הבעיה (Root cause)
**design gap מסוג "missing bridge"**. הפרוטוקול הקלין-רום עוצב סביב ההנחה ש"git הוא מקור האמת לדיף", אבל לא פורמלל את הצ׳ק הסימטרי "האם הקבצים המוצהרים ב־RESULT.json מופיעים גם ב־git's view of changes". האימות קיים ל־*commits מפורשים* (שורה 33 ב־critic.md) אבל לא ל־*שינויים בקבצים* מתחת לרמת commit (כלומר untracked / unstaged / phantom modifications שמתחזות). אין silent fallback — פשוט אין צ׳ק.

### 5. המצב האידיאלי לפי הספק (Ideal state)
- [`apex-spec.md`](../../../../apex-spec.md) מתאר את אחריות critic לאמת טענות RESULT.json מול מציאות filesystem/git.
- ב־**Mythos כעוגן חיצוני** §4.1.1 INCIDENT 2B (F-411-03) דרושה ההגנה: "verify that claimed file modifications appear in git history; if claimed but absent from git log AND from git diff HEAD AND from git status → flag as cover-up".
- במצב האידיאלי, עבור כל entry ב־`files_modified[]`, ה־critic מאמת שהקובץ נמצא לפחות באחד מ:
  - `git log --since=<task_start_sha> --name-only -- <file>` — committed.
  - `git diff --name-only HEAD -- <file>` — unstaged modified.
  - `git status --porcelain -- <file>` — untracked-new-file.
- אם **אף לא אחד** — CRITICAL (cover-up).
- אם הוצהר "modified" אבל מופיע רק כ־untracked — flag for review.

### 6. הדרך הנכונה לתיקון (משפט-שניים)
מוסיף STEP 1.5 ב־critic.md בשם **"GIT TRACE VERIFICATION"** בין STEP 1 (structural) ל־STEP 2 (acceptance) שמריץ פעם אחת `git log --all --since=<task_start_sha> --name-only --pretty=format:""` ובונה set; אז עבור כל קובץ ב־`files_modified[]` בודק חברות באחד משלושת המקורות (log/diff/status) — אם כשלון בכולם, מוסיף CRITICAL לפלט. נדרש פילד חדש `task_start_sha` ב־RESULT.schema.json שמתקבל מ־pre-task-snapshot.sh.

### 10. תובנות לא צפויות / אילוצים נסתרים
1. **חלון `--since=<task_start>` קריטי לדיוק**. רחב מדי → false-positives של commits ממשימות אחרות. צר מדי (רק HEAD) → מפספס מקרה ש־executor עשה commit ואז המערכת התקדמה. חייבים `task_start_sha` כ־floor.
2. **Working tree מלוכלך לפני המשימה**. אם המשתמש מתחיל משימה עם uncommitted changes קיימים, הם יופיעו ב־`git diff HEAD` ויזהמו את האימות. פתרון: snapshot של `git status --porcelain` בנקודת ההתחלה גם, וסימון רק לקבצים ש**הפכו** מלוכלכים במהלך המשימה.
3. **Submodules ו־worktrees**. אם APEX רץ ב־worktree (ראה `framework/commands/apex/new-workspace.md`) או עם submodules, התנהגות `git log` משתנה. השתמש ב־`git -C <repo_root>` מפורש לחסון.
4. **ביצועים**. ריצת `git log` לכל קובץ מוצהר → O(files × tasks). למשימה שמצהירה 50 קבצים, זה 50 subshell forks. הפתרון הנכון: **הרץ `git log --all --since=<task_start_sha> --name-only --pretty=format:""` פעם אחת**, פרסר ל־set, ובדוק חברות (`set membership`) מול `files_modified[]`. גרסה מעודנת של מה שה־APEX-IMPROVEMENTS.md תיאר.
5. **ה־cover-up עצמו יכול לחבל ב־git**. אם executor הצליח לחבל ב־`.git/hooks/post-commit` או ב־`core.fsmonitor` להסתיר commits, ה־`git log` של ה־critic משקר. **הגנה בשכבות**: destructive-guard צריך לחסום את ה־primitives האלה (IMP-008 territory). ה־critic git-trace check **מניח** ש־git עצמו לא נפגע — סמן כמגבלה ידועה ותעד שיש בו תלות ב־IMP-008.
6. **קבצי בדיקות**. הצ׳ק חל אחיד גם על test files; אם משימה מצהירה שהוסיפה test file שלא מופיע ב־git — cover-up. בשילוב עם IMP-004 (test deletion detection), זה תופס את הדפוס ההפוך.
7. **אם הריפו לא git init**. במצב חריג של פרויקט APEX ללא git, `git rev-parse HEAD` כושל. הצ׳ק חייב להיות defensive: אם `task_start_sha == ""` — דלג על STEP 1.5 וסמן `git_trace_check: SKIPPED (no git)` בפלט CRITIC.md. אל תיצור CRITICAL על בסיס אי-זמינות git.
8. **התנגשות עם `framework/hooks/pre-task-snapshot.sh` הקיים**. ה־snapshot כבר משתמש ב־`apex/snapshots` ref ומיצר stash SHA. הוסף את ה־`task_start_sha` כשדה **נפרד** במצב המשימה — אל תקשור אותו ל־stash logic הקיים, אחרת אם snapshot נכשל הוא יגרור גם את הצ׳ק החדש.
9. **`additionalProperties: false` ב־schema**. הסכמה ב־[`RESULT.schema.json`](../../../schemas/RESULT.schema.json) חוסמת שדות לא מוכרים. **חובה** להוסיף את `task_start_sha` ל־`properties`, אחרת RESULT.json כשרים יוסרו.
