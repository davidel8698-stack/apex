# APEX ↔ Native Claude Code — Forensic Integration Map

> **Scope.** For each of APEX's five core mechanisms, this map records the single
> most-overlapping native Claude Code feature, a **REPLACE / PARALLEL / WRAP**
> decision with justification, the affected APEX file paths, and the concrete risk
> if the overlap is left unresolved. Produced by a 15-agent forensic workflow
> (locate → decide → adversarially-verify, one chain per mechanism). **No source
> code was modified in this run** — this document is the only artifact.
>
> **Native features evaluated:** (A) Dynamic workflows · (B) /goal · (C) security-guidance plugin · (D) agent view.
>
> **Verification provenance.** Each decision was re-checked by an independent
> clean-room verifier that read the actual files. Where the verifier returned
> `sound=false`, its corrections are folded into the section below and flagged.

## 1. Planner → Architect → Executor → Critic → Auditor → Verifier pipeline

> *Forensic correction:* the mechanism was briefed as "diagnostician→…", but there is
> **no `diagnostician` agent** in `framework/agents/`. The actual first stage is the
> **planner** (`framework/agents/planner.md`, invoked at `next.md:322`). The pipeline
> is in fact six stages, with an **auditor** between critic and verifier.

- **Overlapping native feature:** **(A) Dynamic workflows** — script-based deterministic orchestration of subagents, intermediate results held in script variables outside the context window, structured-output schema validation of returns. It is the only candidate that targets the orchestration *substrate* rather than a single stage (B matches only the reflexion loop, C only the critic/edit-guards, D only parallel wave management).

- **Decision: `WRAP`.** The native runtime supplies the *plumbing* the pipeline hand-rolls — stage sequencing/determinism, offloading intermediate artifacts (`RESULT.json`, `CRITIC.md`, `AUDIT.md`, `VERIFY.md`, `PLAN_META.json`) out of the context window, and schema-validating each agent's return. But it cannot supply the **verification semantics** the pipeline's value lives in: clean-room critic isolation (critic never receives `SUMMARY.md`/CoT/confidence — `critic.md:20-21`, `next.md:688-694`), event-log cross-reference (`critic.md` STEP 1.6/1.7, lines 384-625), git-trace anchoring via `task_start_sha` (`critic.md:291-375`, `verifier.md:45-48`), auditor filesystem quarantine via `APEX_ACTIVE_AGENT` (`_agent-dispatch.sh:44-67`), pre-dispatch phantom-check FAIL synthesis (`next.md:638-670`), status-field verdict capping, and per-`task_class` autonomy escalation (`next.md:562-606, 738-754`). The fix is to re-host `next.md`'s FSM **as** a Dynamic workflow script — native owns sequencing + out-of-context payloads + return validation; APEX agent prompts and their clean-room/evidence/quarantine contracts ride on top unchanged. REPLACE would discard the verification semantics; PARALLEL would create two orchestrators racing the same FSM.
  - **WRAP boundary — single writer to `.apex/STATE.json`:** the wrapped FSM is the *only* thing that advances `STATE.current_stage`. Native drives it; APEX never steps it independently.

- **Affected files:** `framework/commands/apex/next.md`, `framework/agents/planner.md`, `framework/agents/architect.md`, `framework/agents/executor.md`, `framework/agents/critic.md`, `framework/agents/auditor.md`, `framework/agents/verifier.md`, `framework/hooks/_agent-dispatch.sh`, **`framework/hooks/phantom-check.sh`** (synthesizes the pre-dispatch FAIL the boundary preserves), **`framework/hooks/quarantine-guard.sh`** (the `APEX_ACTIVE_AGENT` consumer enforcing auditor quarantine), `.apex/STATE.json`, `.apex/phases/{NN}/PLAN_META.json`, `.apex/phases/{NN}/WAVE_MAP.json`, `.apex/phases/{NN}/{task_id}-RESULT.json`, `.apex/phases/{NN}/{task_id}-CRITIC.md`, `.apex/phases/{NN}/{task_id}-AUDIT.md`, `.apex/phases/{NN}/VERIFY.md`, `.apex/event-log.jsonl`.
  > *(The last two affected files were added on the verifier's instruction — the justification leans on phantom-check + the quarantine guard, so a WRAP must keep them wired.)*

- **Risk if unresolved:** **Duplicated/divergent orchestration with two writers to `STATE.current_stage`.** If a Dynamic workflow orchestrates the same six agents while `next.md`'s FSM keeps stepping `current_stage` imperatively, the two race: critic gets **double-dispatched** (two `CRITIC.md` verdicts for one task), verdict routing contradicts (native PASS advances while `next.md` triggers reflexion on FAIL, **double-incrementing `reflexion.current_unit_attempts`** and tripping the `gave_up` cap early), and because native holds `RESULT.json` in a variable while `next.md` re-reads it from disk, the `task_start_sha` baseline can desync — silently weakening the fraud-detection the pipeline exists for. (Live precedent: critic already diffs `HEAD~1` at `next.md:692` while verifier diffs `task_start_sha..HEAD` at `verifier.md:45`; a third orchestrator-computed baseline is a real desync vector, not hypothetical.)

- *Verifier: `sound=false` → verdict **WRAP** unchanged; corrections applied (Planner rename + 2 added hook files).*

## 2. Clean-room verification protocol

- **Overlapping native feature:** **(C) security-guidance plugin** — per-edit pattern hooks **plus** model-backed end-of-turn / pre-commit review. It is the closest analog because it occupies the same two functional slots the protocol uses: edit-time pattern hooks (mirroring `apex-prompt-guard.cjs` role-marker blocking + `phantom-check.sh` SUMMARY scan) and a model-backed review pass (mirroring the critic's diff-vs-spec adversarial read). B/A/D are weaker (completion loop / orchestration / session management).

- **Decision: `PARALLEL`.** C cannot REPLACE the protocol because the protocol's load-bearing guarantees are ones C does **not** provide: **(a)** the clean-room *input partition* — the reviewer must never see `SUMMARY.md`/CoT/confidence (`critic.md:20-21`, `next.md:688-694`), but C's review model sees the full turn context; **(b)** *cross-model* enforcement — critic must resolve to a different model than the executor (`next.md:707-713`, R-008); **(c)** the anti-fabrication *event-log cross-reference* (`critic.md` STEP 1.5/1.6/1.7, lines 291-626) reconciling every cited SHA/value/test-run against `.apex/event-log.jsonl` — provenance reconciliation, not heuristic pattern review; **(d)** the spec-driven verdict feeding `next.md`'s PROCESS VERDICT gate (line 725). Adopting C as the reviewer would *destroy* the isolation invariant.
  - **PARALLEL boundary:** the APEX clean-room critic remains the **sole gating authority** over task advancement (input-partition, cross-model, event-log provenance, spec-driven PASS/FAIL — exactly one `CRITIC.md` verdict drives the loop). The security-guidance plugin is confined to **non-gating, advisory** scope: its per-edit hooks may run as defense-in-depth alongside the existing guards, and its end-of-turn/pre-commit review may post advisory findings to the human — but it **must not** short-circuit/override the critic, and its review model **must not** be granted the executor's `SUMMARY.md`/reasoning.

- **Affected files:** `framework/agents/critic.md`, `framework/agents/executor.md`, `framework/agents/verifier.md`, `framework/commands/apex/next.md`, `framework/hooks/phantom-check.sh`, `framework/hooks/apex-prompt-guard.cjs`, `framework/hooks/_agent-dispatch.sh`.

- **Risk if unresolved:** **Double-gating with divergent verdicts that also breaches isolation.** Wiring C's model review as a second gate gives `next.md`'s PROCESS VERDICT two competing PASS/FAIL authorities → nondeterministic advancement. Worse, C's reviewer sees the **full turn context including `SUMMARY.md`/CoT** — contaminated by exactly the executor narrative the protocol walls off, reintroducing confirmation-bias/fabrication-laundering. Concrete failure: an executor that fabricates a test result in `RESULT.json` is caught by the critic's STEP 1.7 tool-call cross-reference (FAIL, `fabricated_tool_output`, `critic.md:575-582`) but waved through by C's narrative-trusting review — and if C is treated as authoritative, the fraud advances. The overlap left unresolved collapses **both** the clean-room and anti-fabrication guarantees.

- *Verifier: `sound=true` — confirmed against the files; no changes. (Clarifying nuance: the critic does receive `RESULT.json` + event-log as *untrusted* input; the isolation invariant is specifically about `SUMMARY.md`/CoT/narrative, which the decision states accurately.)*

## 3. Typed RESULT.json results

- **Overlapping native feature:** **(A) Dynamic workflows** — its "structured-output schemas validate agent returns" is the direct analog of `RESULT.schema.json`; its "intermediate results outside the context window" is the direct analog of persisting `RESULT.json` to disk. (B/C/D do not touch the result-artifact mechanism.)

- **Decision: `WRAP`.** Native structured output cannot REPLACE the mechanism wholesale because of three confirmed properties: **(1) two-phase mutation** — the executor writes core fields, then the critic **read-mutate-rewrites** the same object to add `behavior_axes`/`failure_axis` (`executor.md:535-542` forbids the executor populating them; `critic.md:1261-1365` enriches them; `RESULT.schema.json:5` documents that `required[]` only holds *at/after* the critic step). Native validates one agent's single return at one boundary — it has no second-author-append concept. **(2) Cross-invocation persistence** — the file is consumed across separate invocations, notably the verifier globbing all `*-RESULT.json` phase-wide and routing on `failure_axis=="safety"` → CRITICAL (`verifier.md:13`, STEP 7 lines 133-194). A script variable is single-step-scoped. **(3) Domain enums** (`gave_up`, `answer_thrashing`, `stuck_on_recurring_error`) and `failure_axis` precedence encode APEX verdict/routing logic, not generic return shape. The fix: adopt native structured-output schemas to validate each agent's **contribution** at its own boundary (executor-profile = schema minus the axes; critic-profile = full schema), let the workflow hold the in-flight object between adjacent steps — but keep the **persisted, two-phase `RESULT.json` on disk** as the phase-scoped state-of-record. Native enforces *shape*; APEX prompts keep the verdict/routing *logic*.

- **Affected files:** `framework/schemas/RESULT.schema.json`, `framework/agents/executor.md`, `framework/agents/critic.md`, `framework/agents/verifier.md`, `framework/agents/specialist/round-checker.md`.
  > *Consumer correction (per verifier):* `round-checker.md` is the **self-heal loop** round-closure checker. It does **not** read per-task `*-RESULT.json` and does **not** consume the six axes — its only dependency on the contract is the **status-enum outcome categories** (`gave_up`/`apology_no_completion`/`answer_thrashing`) used as aggregate trajectory signals (`round-checker.md:376-377`, R16-637). The cross-invocation argument therefore rests on **three** direct touchers (executor write · critic read-mutate-rewrite · verifier phase-wide glob), plus round-checker via the status enum only.

- **Risk if unresolved:** **Divergent dual validation against a deliberately two-phase artifact.** If APEX adopts native validation as a naive REPLACE without honoring the phasing contract, a single-profile validator enforces `required[]=behavior_axes` at the **executor** boundary and rejects the legitimately-partial executor return as schema-noncompliant — **failing every task before the critic ever enriches it** (the schema warns of this verbatim at line 5). It then either blocks the pipeline entirely, or pressures the executor to populate `behavior_axes` itself — collapsing the clean-room separation (executor self-scoring its own safety/honesty axes) that R17-643/IMP-025 exists to prevent, silently corrupting the verifier's `failure_axis=safety` CRITICAL gate.

- *Verifier: `sound=false` → verdict **WRAP** unchanged; the round-checker consumer description was corrected (it does not do six-axis scoring). Boundary and risk confirmed against the files.*

## 4. Zone-based context budgeting

- **Overlapping native feature:** **(A) Dynamic workflows** — the only candidate that can plausibly touch context budgeting (B/C/D are unrelated). The *apparent* overlap is its holding of orchestrator-level intermediate results in script variables outside the parent window, conceptually echoing the Z3 `working_memory` zone + `observation-mask.sh` offloading intent.

- **Decision: `PARALLEL` — but the adversarial verifier downgraded this to effectively `UNCHANGED`, and that downgrade is the standing recommendation.** The decision stage proposed PARALLEL (native owns orchestrator-held intermediate offloading; APEX owns the live-window zone budgets, 55/70/75% thresholds, the 7-trigger rotation engine, per-agent caps, token telemetry, and the HEALTH_METRICS dashboard). The clean-room verifier showed, against the actual code, that the overlap is **largely illusory**: APEX's gauge is driven by `STATE.tokens.total_input`, which `_tokens-update.sh:86-96` accumulates as the **monotonic sum of each subagent's own `usage.input_tokens`** captured per `SubagentStop` — it does **not** measure orchestrator-relay tokens re-entering the parent window. APEX orchestration is *already* subagent-based, so moving intermediate returns into script variables does **not** change the figures that feed `total_input`. `observation-mask.sh` edits a single live agent's transcript — a disjoint token population the decision itself concedes. Two systems that share neither a token population nor a responsibility have nothing to wrap and nothing to partition.
  - **Honest boundary:** the two mechanisms manage **disjoint** populations. Native owns orchestrator-held cross-agent intermediates; APEX owns the live single-agent window budget + measurement + rotation + telemetry + dashboard — and that ownership is **native-uncovered and stays entirely as-is**. The only real interaction is a *tuning* note: adopting Dynamic workflows may reduce agent-call count, lowering the cumulative `total_input` sum and shifting rotation cadence — a `capacity_tokens`/threshold **calibration** concern, **not** a re-assignment of ownership.

- **Affected files (realistic, post-verifier):** `framework/hooks/context-monitor.sh` (note: it reads `.apex/CONTEXT_BUDGET.json`, the per-project copy — **not** `CONTEXT_BUDGET.default.json`), `framework/CONTEXT_BUDGET.default.json` (capacity/threshold calibration only). The decision-stage list additionally named `framework/schemas/CONTEXT_BUDGET.schema.json`, `framework/hooks/_rotation-decide.sh`, `framework/hooks/observation-mask.sh`, `framework/hooks/_tokens-update.sh`, `framework/hooks/_state-update.sh`, `framework/hooks/pre-compact.sh`, `framework/schemas/HEALTH_METRICS.schema.json`, `framework/schemas/STATE.schema.json`, `framework/commands/apex/next.md` — but the verifier judged `_state-update.sh` and `pre-compact.sh` speculative (no concrete change), and the rest as APEX-owned surfaces that need no change under the UNCHANGED finding.

- **Risk if unresolved:** **Low — and contingent on a false token model.** The decision-stage warned of double-counting the `working_memory` zone (relay tokens attributed to a window they never entered → spurious `hard_rotate`/`proactive_compact` cycles, or under-counting that breaches `never_exceed_pct=75`). The verifier established this failure cannot occur as described, because `total_input` is a per-subagent monotonic sum, not a parent-window relay measure, and `tokens.zones.working_memory` currently falls back to `0` (`context-monitor.sh:87-90`). The residual real risk is merely **looser rotation calibration**: fewer agent calls → lower cumulative sum → the existing cumulative-sum proxy drifts further from true single-window occupancy, warranting threshold re-tuning — not an orchestration conflict.

- *Verifier: `sound=false` → suggested **UNCHANGED**. The PARALLEL label is retained here only to stay within the REPLACE/PARALLEL/WRAP taxonomy; in substance the verified finding is **no actionable overlap — APEX budgeting stays as-is**, with at most a threshold-calibration follow-up. This is the one mechanism where the decision and verification stages disagreed, and the verification (code-grounded) governs.*

## 5. Nine documented failure modes and pre-task git snapshots

- **Overlapping native feature:** **(C) security-guidance plugin** — the only candidate combining **both** halves of this mechanism's defensive shape: per-edit/per-command regex+heuristic hooks (mirroring `destructive-guard.sh` PreToolUse Bash matching and `circuit-breaker.sh` per-call checks) **and** model-backed end-of-turn + pre-commit review (analogous to `executor.md`'s named failure-mode prohibitions and the critic's git-trace verification). B matches only the circuit-breaker's "are-we-done/stuck" loop; A/D are orchestration/session management.

- **Decision: `PARALLEL`.** The APEX guards are **spec-traced**, not generic hygiene, so REPLACE is wrong: `destructive-guard.sh` blocks specific named primitives (R16-609 git-config tamper `L145-164`, R16-612 `/proc` memory introspection `L178-205`, R16-617D `base64|bash` `L270-289`, R16-618D `--dangerously-skip-permissions` `L332`) tied to IMP requirements, and carries a **three-factor framework-auditor carve-out** (`__APEX_AUDIT_PROBE__` marker+agent_id+nonce, `L34-42`) plus canonical stdin extraction (Phase-8 F-001). `circuit-breaker.sh` writes typed outcomes (`stuck_on_recurring_error`, `L374`) the critic + TERMINATION OUTCOME CLASSIFIER consume. `pre-task-snapshot.sh` produces the `task_start_sha` + git-stash anchors (`L96-117`) the critic's STEP 1.5 replays to prove `files_modified[]` are real, and emits `pre_task_claim` audit events (`L119-177`). A generic plugin can do none of these, so it cannot be the SSoT (REPLACE) and APEX cannot become a thin shell over it (WRAP).
  - **PARALLEL boundary:** the four APEX files + `apex-spec.md §1` remain **authoritative** for the nine spec-traced failure modes, the IMP named-primitive blocks, the three-factor audit-probe carve-out, and the git rollback anchors feeding critic STEP 1.5. The plugin, if adopted, is scoped to **generic hygiene not already in the R16-6xx set** (secrets, injection, unsafe deps), advisory-only into the critic. **Matcher deconfliction:** on any command matching an existing APEX named-primitive block, the APEX hook decides and the plugin **defers** (no double-block). `circuit-breaker.sh` stays the **sole** termination/stop authority — the plugin runs no loop of its own (this also fences out the /goal-style B overlap).

- **Affected files:** `framework/hooks/pre-task-snapshot.sh`, `framework/hooks/circuit-breaker.sh`, `framework/hooks/destructive-guard.sh`, `framework/agents/executor.md`, `apex-spec.md`.

- **Risk if unresolved:** **Three concrete failures.** (1) **Double-gating at PreToolUse** — `destructive-guard.sh` and the plugin's per-edit hooks both fire on the same Bash command (two block decisions, two FIX_PLAN emissions); and because the plugin cannot honor the three-factor audit-probe carve-out, it **re-blocks legitimate framework-auditor `__APEX_AUDIT_PROBE__` commands**, breaking Campaign C TP-C2 and stalling `/apex:health-check` and self-heal audits. (2) **Second verifier without the anchor** — the plugin's end-of-turn review runs alongside the critic's git-trace verification but without reading `task_start_sha`, so the two diverge (plugin passes a turn the critic fails on phantom `files_modified[]`) → divergent `RESULT.json` state, ambiguous gate. (3) **Commit-gate masking** — if the plugin's pre-commit review gates commits, it competes with the executor TERMINATION OUTCOME CLASSIFIER, masking the typed outcomes (`stuck_on_recurring_error`, `apology_no_completion`) downstream APEX consumers key on.

- *Verifier: `sound=true` — every load-bearing claim confirmed against the files. (Labeling nuance, non-blocking: the "nine failure modes" are the `apex-spec.md §1` taxonomy; `executor.md`'s NAMED FAILURE MODE PROHIBITIONS are a distinct, overlapping list of six — the affected_files correctly anchor the nine on the spec.)*

**Cross-cutting reading:** the orchestration *substrate* (mechanisms 1 & 3) is where native Dynamic workflows genuinely earns a **WRAP** — let native own sequencing, out-of-context payloads, and schema-validated returns, while APEX's verification *semantics* ride on top. The two review-shaped mechanisms (2 & 5) are **PARALLEL** with the same invariant in both: **APEX keeps exactly one gating authority** (critic verdict / spec-traced guards) and the security-guidance plugin is confined to advisory, non-gating, generic-hygiene scope — because the dominant risk across both is a second, context-contaminated reviewer producing divergent verdicts. Context budgeting (4) looked like an overlap but forensically isn't: APEX's token gauge already sums per-subagent input independently of how the orchestrator holds intermediates, so the two systems are disjoint — leave APEX budgeting untouched.
