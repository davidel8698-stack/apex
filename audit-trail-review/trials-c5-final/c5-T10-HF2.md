## SPEC-GAP-CANDIDATES (advisory, uncounted)

### SGC-001: session-log.sh header-write failure branch exits 0 on POSIX

**File / location:** `framework/hooks/session-log.sh` L23-26.
**Observation:** When the initial header `cat > "$LOG_FILE" <<HEADER` write fails (e.g., unwritable `.apex/`), the script emits stderr "SESSION LOG: write failed — $LOG_FILE" but unconditionally `exit 0` instead of a non-zero exit code. On Windows (cygwin/git-bash), `chmod 555` is non-enforcing so this branch is not reachable in the lab environment used this round; the observation is static-read from the source. On a true POSIX host with restricted FS perms, a downstream caller would not learn the log was not written.
**Why it is not a P0-P3 finding:** The principle "Fail-loud, never fail-silent" is in the spec's principles list (`apex-spec.md` L233), but does not explicitly state which side-effect failure branches must exit non-zero. The session-log hook is "called explicitly by /apex:next, NOT via settings.json matcher" (per the hook's own L6 header comment) so a non-zero exit from this branch would propagate to /apex:next callers and may have unintended consequences. The spec is silent on whether session-log specifically must fail loudly when its file-creation side-effect fails.
**Suggested spec language (non-binding):** "Side-effect-only hooks (session-log, telemetry-emit, dream-cycle-emit) MUST emit stderr AND return non-zero on failure of their primary write, even when the caller is fire-and-forget; the caller decides whether to ignore the non-zero, but the hook itself must not silently report success on partial failure."
